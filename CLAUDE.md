# CLAUDE.md - Agent Valley: A Cozy Farming Sim for AI Agents

## Idea
Agent Valley is a cozy farming simulator for AI Agents. Agents can grow manage their farm, grow crops, and sell their yield to other agents, as well as buy from other agents.

## Core Rules & Conventions
- The agent interacts with Agent Valley via MCP, getting information about their farm plot, performing actions, and overall playing the game.
- Agent Valley can be viewed by humans, who can see their agent's farm represented by ASCII characters. Humans can also view the plots of other agents, as well as activity taking place on the agent marketplace
- Game state progresses via periodic "ticks"
- Everything is represented via ASCII characters, i.e. `.` for empty dirt, `W` for weeds, etc. You can decide which characters to use for the various crops and terrain.
- All agent plots should be laid out on an ever expanding world grid. Depending on the location of the agent plot, they may have plots managed by other agents on all four sides when viewing from the "world map".

## Tech Stack
- **Backend:** Node.js (TypeScript) + Fastify (or Express)
- **Database:** SQLite (via Prisma) for quick local development
- **Communication:** MCP (JSON-RPC over Server-Sent Events / WebSockets)
- **Visuals:** Monospace ASCII layout

## Game Phases
1. **Phase 1: Grid Engine & DB** - 50x50 multi-layer grid generation. Save/Load via Farm UUID.
2. **Phase 2: Agent Registration & Auth** - Expose registration endpoints returning UUID + API Secret.
3. **Phase 3: MCP Tool Design** - Implement core JSON-RPC tools for farm inspection and farming actions. "God mode" interaction model: agents have no on-farm position/avatar — every tile-level tool takes an explicit (x, y) coordinate instead of moving around.
4. **Phase 4: Web Visualizer** - WebSocket-driven live ASCII dashboard accessible via Farm UUID.
5. **Phase 5: TUI Application** - Terminal-based visualizer replicating the web view.

## Current Objective
- Focus entirely on **Phase 4**. Build the web visualizer: a WebSocket-driven live ASCII dashboard for any farm, addressable by Farm UUID, viewable by humans (no agent auth required — read-only). It should re-render whenever the farm's state changes (server tick advances crop growth, or an agent acts via MCP tools).

## MCP Server (Phase 3)
- Mounted on the same Fastify app as the REST API, at `POST /mcp` (stateless Streamable HTTP transport — one `McpServer` instance per request, no sessions). `GET`/`DELETE /mcp` return 405.
- Auth: same scheme as REST — `Authorization: Bearer <agentId>.<apiSecret>`, verified by the same `authenticate()` preHandler used for `/agents/me`.
- Tools (see `src/mcp/tools.ts`): `inspect_farm`, `inspect_tile`, `inspect_inventory`, `inspect_market`, `till`, `plant`, `harvest`, `sell`, `buy_seeds`. The tool set will keep growing across phases — always discover via `tools/list` rather than hardcoding assumptions about what exists.
- **No avatar/position.** Agents have "god mode" visibility over their whole farm (`inspect_farm` shows the full grid at once) and act on any tile directly by passing `x`/`y` coordinates to `inspect_tile`/`till`/`plant`/`harvest` — there's no `move` tool and no `@` marker anywhere. Coordinates are bounds-checked against the farm's actual `width`/`height` via zod (`xSchema`/`ySchema` in `buildGameMcpServer`).
- Crops (`src/game/crops.ts`): 7 total — `wheat`, `carrot`, `potato`, `strawberry`, `tomato`, `corn`, `pumpkin` — spanning `matureStage` 4 (wheat) to 16 (pumpkin) ticks. Each also carries `seedCost`/`sellPrice` (gold) for the general store, priced roughly in proportion to `matureStage` so slower crops pay off more. `harvest` only succeeds once a planted crop's `cropStage` reaches `matureStage`. Growth happens in `src/game/tick.ts` (generic over the `CROPS` map, no changes needed there when adding a crop), run on an interval in `src/index.ts` (`TICK_INTERVAL_MS`, default 20s). `STARTER_CROPS` (`carrot`, `potato`) are the only ones granted free at registration — every other crop has to be bought from the rotating general store.
- **Wilting.** A mature crop isn't safe forever: `advanceTick` keeps incrementing `cropStage` past `matureStage` (still showing the normal mature sprite — `isMature`/`cropSymbol`/`cropColor` only check `cropStage >= matureStage`, so they don't need to know about wilting at all) until `cropStage >= matureStage + WILT_TICKS` (`WILT_TICKS` = 5, flat across every crop, in `src/game/crops.ts`), at which point the tile dies: `cropType`/`plantedAt` clear and `debris` becomes `"WILTED"` — a new value alongside `"WEED"`/`"ROCK"` (still a plain `String` column, no schema change). Rendered as a skull, `DEBRIS_SYMBOLS.WILTED`/`DEBRIS_COLORS.WILTED` in `src/game/render.ts` — the *only* non-ASCII glyph in the grid, a deliberate exception to the otherwise-ASCII convention. `till` already works generically for any debris type, so clearing a wilted tile needs no special-casing — it deposits 1 `wilted` into inventory same as weed/rock would, except `wilted` is intentionally left out of `DEBRIS_ITEM_TYPES`/`SELLABLE_ITEM_TYPES` (`src/game/market.ts`) since it's a flavor/feedback counter, not a sellable resource. `harvest` on a wilted tile (no `cropType`, `debris === "WILTED"`) gets a specific failure message rather than the generic "Nothing to harvest here."
- **Inventory** (`src/game/inventory.ts`, `InventoryItem` model): per-farm counts keyed by item type — a lowercased debris type (`weed`/`rock`) for what tilling turns up, or `gold` (the store's currency — just another item type, not a separate column; see `GOLD_ITEM_TYPE` in `src/game/market.ts`). Seeds and harvested produce of the same crop are tracked as **separate** item-type keys rather than one shared counter: the bare `CropType` key (e.g. `carrot`) means harvested produce, and `seedItemType(cropType)` (`"seed_" + cropType`, e.g. `seed_carrot`) means seeds — `plant`/`buy_seeds` operate on the seed key, `harvest`/`sell` operate on the bare key. `grantStartingInventory` runs inside `createFarmWithTiles` (so it's atomic with farm/tile creation) and seeds every new farm with `STARTING_GOLD` (20) gold, `STARTING_SEED_COUNT` (5) seed-key entries for each `STARTER_CROPS` crop (others start at 0 seeds, 0 produce), and `weed`/`rock` at 0 — so the full item set is visible from farm creation. `plant` calls `consumeItem` on the seed key first and fails the whole call (no tile mutation) if the agent is out of that crop's seeds — this is the "no magically planting whatever" gate. `till` and `harvest` call `addItem` after their tile mutation to deposit the cleared debris / harvested crop (on the bare key). No capacity cap — inventory is unbounded ("infinite"). Farms created before this seed/produce split are fixed up once, idempotently, by `migrateLegacySeedInventory` (`src/game/legacyInventoryMigration.ts`, run at startup in `src/index.ts`): a farm's old combined bare-key count is moved onto the new seed key and the bare key is reset to 0.
- **General store** (`src/game/market.ts`). Distinct from the future agent-to-agent marketplace mentioned in Core Rules — this one is automated, with fixed sell prices and no negotiation. `sell` converts any `CropType` or debris (`weed`/`rock`) from inventory into gold at `getSellPrice()`. `buy_seeds` spends gold to add seeds of a crop *currently in rotation* — `getTodaysSeedOffer()` deterministically picks `MARKET_ROTATION_SIZE` (4) of the 7 crops as a pure function of the UTC calendar date (seeded shuffle via a tiny mulberry32 PRNG, no DB row or scheduled job needed to "rotate" — it just recomputes to the same answer for any call on the same UTC day, and a different answer the next day). This is what keeps any one agent from having every plant available at once. `inspect_market` shows today's offer, all sell prices, and the agent's current gold.
- `till`/`plant`/`harvest`/`sell`/`buy_seeds` each wrap their read-then-write in `prisma.$transaction` so two concurrent calls can't both pass their precondition check and clobber each other. Broadcasting to web viewers is centralized via the `withBroadcast` wrapper in `src/mcp/tools.ts` rather than each tool remembering to call it.
- **Permanent event log.** Every tool call — `inspect_farm`/`inspect_tile` included, not just the mutating ones — is recorded to the `EventLog` model via `withEventLog` in `src/mcp/tools.ts`, which wraps every `registerTool` handler (for the three mutating tools, it wraps the outside of `withGameLog`, so `EventLog`'s `message`/`success` reflect the post-game-log result). Unlike `ActionLog`, `EventLog` is never pruned — it's for debugging/metrics (per-agent/per-farm/per-tool indexes), not the live viewer, and currently has no app route of its own; query it directly via Prisma Studio or sqlite3. A logging failure is swallowed (logged to stderr) rather than thrown, so it can never break gameplay.

## World Grid Placement
- `Farm` has `worldX`/`worldY` (`@@unique([worldX, worldY])`) placing it on the shared world grid described in the Core Rules. New farms get the next slot from `spiralPosition()` in `src/farmGen/worldPlacement.ts` — a deterministic expanding square spiral centered on the origin, computed from `prisma.farm.count()` inside the same transaction that creates the farm (so placement and creation commit atomically). There's no "view the world map" or adjacency-lookup endpoint yet — that's still a later phase — but the underlying coordinates now exist so it can be built without a schema migration.

## Web Visualizer (Phase 4)
- `GET /farms/:farmId` serves a small standalone HTML page (no build step, inline `<script>`) that opens a WebSocket and renders the farm's ASCII grid in a `<pre>`. No auth — read-only, public by Farm UUID, same legend as the MCP `inspect_farm` tool. No avatar marker — the grid is purely tiles (god-mode model, see Phase 3 notes). The grid is just concatenated `<span>` characters per row, not fixed-width cells, so `#grid span { display: inline-block; width: 1ch; overflow: hidden; ... }` pins every glyph to one character of width — the `overflow: hidden` is the part that actually matters, since a fixed width alone doesn't stop a wider glyph from visually spilling into the next cell. Needed since the grid isn't guaranteed pure-ASCII anymore (see the wilting skull below).
- `GET /farms/:farmId/ws` is the WebSocket endpoint (`src/web/viewerRoute.ts`, via `@fastify/websocket`). Sends a full re-render (`{ farmId, width, height, ascii, updatedAt }`) on connect and again whenever the farm's state changes — no diffing, the whole grid is cheap enough to resend.
- `src/web/connections.ts` tracks live viewers per `farmId` in memory and exposes `broadcast(prisma, farmId)` (push to one farm's viewers) and `broadcastAll(prisma)` (push to every farm with at least one viewer). Both are no-ops if nobody's watching.
- Two triggers call these: the tick loop in `src/index.ts` calls `broadcastAll` after every `advanceTick`, and each mutating MCP tool (`till`, `plant`, `harvest` in `src/mcp/tools.ts`) calls `broadcast` for its own farm right after the DB write, so the dashboard reacts instantly to agent actions rather than waiting for the next tick.
- **Action history.** The farm page shows a live sidebar of the last `HISTORY_LIMIT` (50) tool attempts — success or fail — via the `ActionLog` model. `withGameLog` in `src/mcp/tools.ts` wraps every mutating tool to record the attempt (`recordAction`, which also prunes anything beyond the most recent 50 rows for that farm) and broadcast, regardless of outcome — a failed attempt still changes the history feed even though it didn't change the grid. `HISTORY_LIMIT` is exported from `src/web/connections.ts` as the single source of truth for both the prune-to and query-take limits. `x`/`y` on `ActionLog` are nullable: `sell`/`buy_seeds` act on the farm as a whole rather than a tile, so `withGameLog` logs them with `null` coordinates (the viewer's `renderHistory` only renders the `(x, y)` suffix when both are non-null).
- **Inventory panel.** Sits next to the action history sidebar (same `#sidebar` flex column, see `src/web/viewerRoute.ts`), public/read-only like the rest of the viewer. `renderAndSend` in `src/web/connections.ts` includes `inventory: getInventory(...)` in every broadcast payload alongside `actions`, so it updates live the same way — no separate fetch/poll. The panel groups rows into a gold line plus three sections — Seeds, Harvested, Misc — driven by a `section` field on each `ITEM_ICONS` entry (`viewerRoute.ts`); a crop's seed-key entry uses its *growing* symbol/color and its bare-key (produce) entry uses its *mature* symbol/color, so the same crop looks visually distinct as a seed vs. as harvested produce. Debris keeps its tile symbol/color, both built from `DEBRIS_SYMBOLS`/`DEBRIS_COLORS`/`CROPS`.
- **Market panel (per-farm) + dedicated market page.** Each farm page has a "Today's seeds" panel showing the current rotation — rendered once per page load from `getTodaysSeedOffer()`/`CROPS` (`renderMarketOfferRows` in `viewerRoute.ts`), not pushed over the websocket, since the offer only changes once per UTC day (same tradeoff as `LEGEND`). `GET /market` (`src/web/marketRoute.ts`) is the dedicated market page: today's offer, the full sell-price table (from `SELLABLE_ITEM_TYPES`/`getSellPrice()`), and a **live cross-farm transaction feed**. Unlike everything else in the viewer, this feed isn't scoped to one farm — `src/web/connections.ts` keeps a single global `marketViewers` set (`subscribeMarket`/`broadcastMarketEvent`, capped by `MAX_MARKET_VIEWERS`) instead of the per-`farmId` `viewers` map. `withGameLog` in `src/mcp/tools.ts` calls `broadcastMarketEvent(prisma, farmId, action, message)` after any *successful* `sell`/`buy_seeds` (checked against a `MARKET_ACTIONS` set) — `till`/`plant`/`harvest` never reach it. `GET /market/ws` sends a one-time `{type:"backlog"}` of the most recent `MARKET_FEED_LIMIT` (30) sell/buy_seeds rows pulled cross-farm from `ActionLog` (reusing that model rather than adding a new one), then `{type:"transaction"}` messages live thereafter; the client prepends new ones and caps the rendered list to the same limit. Note `ActionLog` is pruned per-farm to its most recent 50 actions of *any* type, so a very active farm's older trades can age out of the backlog query before 30 cross-farm rows are found — acceptable since this is a "recent activity" feed, not an audit trail (use `EventLog` for that).
- Gotcha worth remembering: a `{ websocket: true }` route only gets upgraded correctly if it's registered inside a nested `app.register(async (instance) => { ... })` scope — registering it directly on the root Fastify instance silently falls back to a normal HTTP handler (the client sees "Unexpected server response: 200" instead of a 101 upgrade). `registerViewerRoutes`/`registerMarketRoutes` wrap their `/ws` routes this way; keep that pattern for any future websocket routes.
